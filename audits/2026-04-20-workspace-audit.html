Executive Summary

Your workspace is lean, active, and plan-driven — but carrying ~20% dead weight and 3 structural risks that cost clarity more than credits.

The good news: Usage is real, not theoretical. decisions/log.md is live (32 entries), priorities.md was updated today, and every single plan in plans/ has shipped or is executing. No ghost plans. Hooks auto-push at session end and it works. Core tools (Footage Organizer, Content Pipeline, Invoice System) show recent touches.

The problems: 4 overlapping Python projects (photo-organizer vs footage-organizer, creator-intel vs content-researcher, expense-tracker vs invoice-system, screen-recording-analyzer vs content-pipeline). 4 dead web prototypes. 6 of 8 workflow docs describe code that's 1–3 versions behind reality. VPS has dual-scheduling (cron AND systemd) for the same jobs — that's a real outage risk. ui-ux-pro-max skill is so broadly triggered it will hijack project-specific skills. And the Anthropic API cache is not being used anywhere.

The bottom line: Money-wise you're already running lean (total recorded spend < $0.01). The real prize of this audit is clarity: fewer moving parts, docs that match code, one source of truth for project status, and zero-risk scheduling on the VPS.

VPS · Deploy layer

Highest-risk finding in the audit. The VPS has both cron and systemd trying to schedule the same jobs. This is a real outage vector — not theoretical.

  • ⚠
    morning-briefingsystemd service passes --schedule (internal scheduler) AND cron fires it at 8am. Two schedulers, one job. Risk of double-runs or race conditions.
  • ⚠
    invoice-scansystemd service says --time 12:00 (noon). Cron job says 9am. Which is canonical? Currently both can fire.
  • ⚠
    email-agentService file uses Restart=always + 15s. vps-setup.sh builds a service that uses Restart=on-failure + 30s. File drift between committed config and install script.
  • ⚠
    email-agent duplicate runsPer memory, Gray already suspects email-agent is running on both Mac and VPS simultaneously. Confirm which is canonical and shut the other down.
Recommendation: pick one scheduler — cron for time-based jobs (8am briefing, 9am invoice), systemd only for the always-on email-agent daemon. Delete the conflicting .service files. Re-run bash deploy/vps-setup.sh from clean state.
